01Overview & what was built

Starting point: tesbih — a static HTML + CSS + JavaScript educational app with 3 languages, themes, quiz, and narrator. No frameworks, no dependencies. Target: a real Android app in the Google Play Store.

Imagine a website that acts like an app. You want it on people's phones, not just in a browser. That's what we did: we put the website inside a tiny box that Android knows how to open, so it feels like any other app.

08Anatomy of an Android app (newbie tour)

What actually IS an Android app? What does "compile" mean? What's inside the APK? This section explains every piece of the android/ folder — in plain language, with analogies a 10-year-old can follow.

An Android app is basically a zipped-up shoebox. Inside the shoebox: your app's pictures (icons, splash), a rulebook about what the app is allowed to do (manifest), translated texts (for different languages), and a compiled version of the code the phone can run fast. You don't ship the raw source — you ship the packed shoebox.

1 · Mental model — what is an Android app?

📦 An APK is a ZIP file

Rename app-debug.apk to app-debug.zip → double-click → it extracts. Android just gave the extension a special name. Inside: a standardised folder structure the phone knows how to install.

🧬 An AAB is a smarter ZIP

When uploaded to Play Store, Google unzips the AAB and makes a custom APK for each phone (only the icons at your screen density, only your language, only your CPU architecture). User downloads smaller files. Same source, smarter packing.

📜 Every app needs a "manifest"

The AndroidManifest.xml file is like a passport: app name, icon, permissions requested, which screen to open first, what devices it supports. Android refuses to install without one.

✍️ Every app must be signed

Signing = attaching a cryptographic fingerprint that proves "the same author made this". Android uses the signature to recognise updates. Lose your signing key → the phone treats your new version as a different app entirely.

4 · What "compiling" actually means

Compiling = translating. Your code is written for humans. The phone's brain is a special chip that only reads a different language (machine code). "Compile" is the translation step. Our web app (HTML/CSS/JS) isn't compiled — it's carried along as-is and interpreted at runtime by the WebView, like a book you read aloud instead of memorising.

What happens when you run ./gradlew assembleDebug:

  1. Gradle reads the build scripts — settings.gradle, build.gradle, app/build.gradle. Figures out what to build and in what order.
  2. Downloads dependencies (first run only) — Capacitor runtime, AndroidX libraries, etc. Cached to ~/.gradle/caches/.
  3. Compiles Java source (MainActivity.java) → .class files (JVM bytecode). The javac compiler.
  4. Compiles Kotlin source (none in our project, but Capacitor's plugins may use it).
  5. Dexes — converts .class files to .dex (Dalvik Executable, Android's optimised bytecode format). The phone's ART runtime loads this.
  6. Compiles resources — XML files (layouts, strings) get converted to a compact binary format. Images get optimised. The aapt2 tool does this.
  7. Packages — zips up classes.dex + resources.arsc + res/ + assets/ into a single .apk file.
  8. Signs — attaches a cryptographic signature using the debug keystore (or your release keystore).
  9. Aligns — rearranges bytes inside the APK so the phone can memory-map them efficiently (zipalign).

7 · Signing — the lock on the shoebox

Every APK/AAB must be signed. The signature proves "this build came from the same author as previous builds".

🔓 Debug keystore

Android ships with a default keystore at ~/.android/debug.keystore. Everyone's Debug keystore is different but Play Store rejects them all. Use only for local dev & sideload.

🔒 Release (upload) keystore

YOU generate this with keytool -genkey. Protect it like a password. Play Store recognises YOUR signature as yours forever. Lose it → you can never update this app again.

☁️ Play App Signing

Since 2020, Play Store offers to hold the MASTER signing key for you. You upload builds signed with your upload key; Google re-signs them with the true distribution key. Benefit: if you lose your upload key, Google can reset it.

📝 What's actually signed

The entire APK (minus the signature files themselves). Any change — one byte of HTML, one pixel of an icon — invalidates the signature and must be re-signed.

8 · What adb install actually does

  1. USB bridge — adb opens a TCP-over-USB channel to the adbd daemon running on the phone (which is why USB debugging must be on).
  2. APK transfer — the file is streamed from your Mac to /data/local/tmp/ on the phone.
  3. Package Manager — pm install runs on the phone. It:
    • verifies the APK signature
    • checks permissions requested in the manifest
    • extracts classes.dex to /data/app/org.workshopdiy.tesbih/
    • optimises the DEX with the phone's ART runtime (creates .oat / .vdex files)
    • copies icons to the right system caches
    • registers the app with the launcher → icon appears
  4. First launch — tapping the icon sends an Intent to MainActivity; Android allocates memory, spawns a process, runs the activity's onCreate. Our MainActivity just inherits from Capacitor's BridgeActivity, which creates a WebView and loads file:///android_asset/public/index.html. From there, it's your web app.

Recap for workshop: You wrote a web app (HTML + JS + CSS). Capacitor put it in a shoebox (APK). Android Studio compiled a tiny bit of Java (5 lines, really) that opens a browser window inside the app. Play Store distributes the shoebox. The phone unpacks it and adds an icon. That's the whole journey.

09Anatomy of an iOS app (newbie tour)

Mirror of §08, but for iPhone. What's an .ipa really? What does Xcode actually do? Why is iOS signing so much more complicated than Android? Everything in plain language.

An iPhone app is a shoebox too, but the post office (Apple) is picky. You can't just hand them any shoebox — you need a special sticker (certificate) that only Apple issues, and a stamp (provisioning profile) that says exactly which shoeboxes can go to which addresses (iPhones). That's why iOS development feels more ceremonial than Android.

1 · Mental model — what is an iOS app?

📦 An IPA is a ZIP file

Like APK on Android. Rename MyApp.ipa → MyApp.zip → extract → you'll find a Payload/MyApp.app/ folder inside. The .app isn't a file — it's a Finder-disguised directory.

📁 A .app is a bundle

Bundles are just folders with a special extension (.app, .framework, .bundle). Finder shows them as single icons but ls reveals the real contents. Universal concept across macOS & iOS.

📜 Info.plist is the manifest

Instead of AndroidManifest.xml, iOS uses Info.plist — a key-value file (XML or binary) listing: bundle ID, display name, version, permissions descriptions, supported orientations, minimum iOS version.

🔏 Code signing is mandatory

Unlike Android (which tolerates debug keystores), iOS won't run any code unless it's signed by a recognised Apple certificate. Even on your own iPhone. Even in Simulator (but the Simulator cert is automatic).

4 · What "compiling" means on iOS

iOS builds compile more things than Android. Your app icons, your storyboards, your asset catalogs — all get pre-processed into compact binary formats Apple's runtime can load faster. Your Swift code becomes native ARM64 machine code directly (no bytecode layer like Android's DEX).

What happens when you hit the ▶ Run button in Xcode (or xcodebuild build):

  1. Xcode reads the project — project.pbxproj, Podfile, all .xcconfig files.
  2. Resolves Swift Package Manager & CocoaPods (incremental — only if something changed).
  3. Compiles Swift — AppDelegate.swift, SceneDelegate.swift, Capacitor's Swift files → .o object files → ARM64 machine code (for device) or x86_64/ARM64 (for simulator). Uses the swiftc compiler.
  4. Compiles Objective-C — some Capacitor plugins are still Obj-C → same .o format.
  5. Compiles storyboards — Main.storyboard and LaunchScreen.storyboard → .storyboardc directory (binary plist).
  6. Compiles asset catalog — Assets.xcassets → a single Assets.car file (optimised image blob).
  7. Processes Info.plist — preprocessor runs, variables substituted, converted to binary plist.
  8. Links — all .o files + frameworks → single Mach-O executable named App.
  9. Copies resources — our public/ web files, Assets.car, storyboards, embedded provisioning profile — all copied into App.app/.
  10. Signs the bundle — codesign tool attaches the signature using your Apple developer certificate + embeds the provisioning profile.
  11. Validates — Xcode runs codesign --verify, checks entitlements match provisioning profile.

7 · Signing & provisioning — the hard part

Apple requires four things for every signed build, all of which Xcode can usually auto-generate if you click "Automatically manage signing":

1. Developer certificate

A public/private key pair issued by Apple. The private key lives in your Mac's Keychain; the public key sits in Apple's servers. Tied to your Apple ID / team.

Types: Apple Development (for dev builds) · Apple Distribution (for App Store + TestFlight)

2. App ID

A bundle identifier (e.g. org.workshopdiy.tesbih) registered in your Apple Developer account. Must exactly match the CFBundleIdentifier in Info.plist.

3. Provisioning profile

A file Apple signs that lists: this app ID × this certificate × these devices (or "App Store"). Embedded into the IPA as embedded.mobileprovision. The iPhone's OS checks it before allowing launch.

Types: Development · Ad Hoc · App Store · Enterprise

4. Entitlements

A file listing which capabilities the app uses (Push, HealthKit, iCloud, etc.). Must match what the provisioning profile allows. For our simple Tesbih: no special entitlements.

In Xcode → target → Signing & Capabilities tab, tick "Automatically manage signing" and pick your Team. Xcode requests the cert from Apple, creates the app ID, generates the profile, and embeds everything. The only time you touch this manually is for enterprise / multi-team / CI scenarios.

Free Apple ID limitation: You can sign & install on your own device without paying $99, but the certificate expires in 7 days and you can't upload to App Store Connect or TestFlight. Paid membership → 1-year certs + App Store access.

8 · What happens when installing

🖥️ Via Xcode (dev build)

  1. Xcode builds + signs with your dev cert
  2. Transfers bundle to device via USB
  3. iOS checks signature + provisioning profile
  4. On first install: iPhone asks you to trust the dev cert (Settings → General → VPN & Device Management)
  5. App installs, icon appears

✈️ Via TestFlight

  1. You Archive + Upload to App Store Connect
  2. Apple re-signs with their distribution cert
  3. Build "processes" for 5–30 min
  4. Testers get an email invite
  5. Tester opens TestFlight app → taps Install
  6. App installs like any App Store app

🏪 Via App Store (after review)

  1. Apple reviews (24–48h typically)
  2. If approved, build moves to "Ready for Sale"
  3. You (or Apple automatically) release it
  4. Appears in App Store search globally
  5. Anyone can install

🔐 What the phone verifies on launch

  1. Signature chain: cert → Apple root CA
  2. Cert not revoked
  3. Provisioning profile not expired
  4. Device UDID in profile (dev/ad-hoc only)
  5. Entitlements match profile
  6. If all pass → spawn process, call UIApplicationMain

Recap: iOS apps are stricter because Apple controls both the hardware and the store. Every app that ever runs on any iPhone has been signed by a cert that traces back to Apple, and Apple decides who gets certificates. That's why the Play Store review is 2–7 days and App Store is 24–48h — Apple pre-filters developers at the door, Google pre-filters apps at the checkout.

12The manage.sh script (full reference)

A single entry point that auto-detects your OS (macOS / Linux / Windows git-bash / MSYS2) and launches the right platform-specific menu. Designed so nobody has to memorise commands and so the same instructions work across all three operating systems. Run it, pick a number, follow prompts.

Think of it as the control panel of a spaceship. You don't need to remember which lever does what — it's labelled, and it asks before doing anything dangerous. And it speaks whatever language the spaceship is running on (macOS / Linux / Windows), so you only learn it once.

There's only one command to remember, regardless of OS: ./manage.sh. The dispatcher inspects uname -s and $MSYSTEM, then execs the matching platform script.

./manage.sh

FORCE_OS=linux ./manage.sh
FORCE_OS=macos ./manage.sh
FORCE_OS=windows ./manage.sh

16BLE & micro:bit apps (Web Bluetooth → Capacitor)

For wrappers around web apps that pair with BBC micro:bit (or any BLE device) over Web Bluetooth, the WebView needs a translation layer. This section explains why and how.

16.1 Why Web Bluetooth doesn't "just work" in Capacitor

Open your micro:bit web app in Chrome on a phone — the "Connect" button works because Chrome exposes navigator.bluetooth.requestDevice(...). Wrap that same app with Capacitor and install the APK — the same button silently fails. Why?

Capacitor uses Android's System WebView (the same Chromium engine, but stripped of some "browser-only" APIs). Web Bluetooth is hard-disabled in System WebView for security reasons (a random app shouldn't be able to silently pair with nearby BLE devices). There's no flag, no manifest tag, no Capacitor option to turn it back on. The API is genuinely absent: navigator.bluetooth is undefined.

16.2 The fix: Capacitor BLE plugin + a shim

Two pieces, one transparent solution:

  1. Native plugin — @capacitor-community/bluetooth-le exposes a JS API that bridges to native Android Bluetooth. Different shape than Web Bluetooth, but it has all the same capabilities: scan, connect, services, characteristics, read, write, notifications.
  2. Web Bluetooth shim — www/web-bluetooth-shim.js monkey-patches navigator.bluetooth so it forwards calls to the Capacitor plugin. The web app's existing code keeps using navigator.bluetooth.requestDevice(...); the shim quietly translates each call.

Net effect: the web app needs zero changes. The wrapper handles it.

16.3 What's wired up at spawn time

When books.yaml entry has needs_ble: true, koutoub-spawn.sh automatically:

  1. Copies www-overlay/web-bluetooth-shim.js into the wrapper's www/
  2. Injects three <script> tags into www/index.html (after <head>): 
    <script src="cordova.js"></script>
<script src="cordova_plugins.js"></script>
<script src="web-bluetooth-shim.js"></script>
  3. Adds @capacitor-community/bluetooth-le to package.json dependencies
  4. Drops _patch_ble.py into the wrapper root for use by the build menu

The first time you run ./manage.sh → option 4 (sync), the menu detects the shim, runs npx cap add android, and then _patch_ble.py patches android/app/src/main/AndroidManifest.xml:

<uses-permission android:name="android.permission.BLUETOOTH_SCAN"   android:usesPermissionFlags="neverForLocation" tools:targetApi="s" />
<uses-permission android:name="android.permission.BLUETOOTH_CONNECT" />
<uses-permission android:name="android.permission.BLUETOOTH_ADVERTISE" />
<!-- pre-Android 12 fallback -->
<uses-permission android:name="android.permission.BLUETOOTH"          android:maxSdkVersion="30" />
<uses-permission android:name="android.permission.BLUETOOTH_ADMIN"    android:maxSdkVersion="30" />
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" android:maxSdkVersion="30" />
<uses-feature android:name="android.hardware.bluetooth_le" android:required="true" />

16.4 The pairing flow on the phone

  1. App launches → splash → main UI loads → shim installs over navigator.bluetooth
  2. User taps a "Connect" button in the web app
  3. App calls navigator.bluetooth.requestDevice({ filters: [{ services: ['e95d...'] }] })
  4. Shim catches the call, runs BleClient.initialize() on first call, then BleClient.requestDevice(...)
  5. Android shows a system-level device picker dialog — the user picks their micro:bit by name
  6. Shim returns a fake BluetoothDevice object whose gatt.connect() forwards to BleClient.connect(deviceId)
  7. The web app calls service.getCharacteristic(...), characteristic.startNotifications(), characteristic.addEventListener('characteristicvaluechanged', ...) — all forwarded by the shim
  8. BLE notifications fire characteristicvaluechanged events, the web app updates its UI, life is good

16.5 What the shim does NOT cover

  • Descriptors (BluetoothRemoteGATTDescriptor) — rarely used, not implemented
  • Advertisement watching (watchAdvertisements) — not implemented
  • Paired devices list (navigator.bluetooth.getDevices()) — not implemented
  • Real-time connection state — server.connected reflects last-known state, not a live signal

If your app uses any of those, you need to either patch the shim or migrate that part of the app's code to the Capacitor plugin's API directly.

16.6 Permissions UX on Android 12+

On Android 12 (API 31+), Bluetooth permissions are runtime permissions — the user must approve them in a dialog when the app first tries to scan or connect. The Capacitor BLE plugin's initialize() handles the prompt automatically. Make sure the user taps "Connect" (a clear gesture) BEFORE any scan starts — Android requires a user gesture to grant BLE permissions in some configurations.

Pre-Android 12 (API ≤ 30), Bluetooth scanning required ACCESS_FINE_LOCATION for historical reasons — yes, scanning for BLE used to count as "location data". The manifest declares this with android:maxSdkVersion="30" so it's only requested on older devices and skipped on Android 12+.

1.Install the debug APK on a real Android phone (BLE doesn't work in headless emulators).

4.Approve Bluetooth permissions in the system dialog.

16.8 Debugging recipes

Probe the shim is installed — connect Chrome devtools to chrome://inspect/#devices and run in the WebView console:

typeof navigator.bluetooth                  // "object"  ← shim installed
typeof navigator.bluetooth.requestDevice    // "function"
typeof window.Capacitor.Plugins.BluetoothLe // "object"

If navigator.bluetooth is undefined the shim didn't load. Check the Network tab for a 404 on web-bluetooth-shim.js, and verify cordova.js + cordova_plugins.js are loading first (they bring in the Capacitor runtime that the shim depends on).

Verify permissions are declared:

grep BLUETOOTH android/app/src/main/AndroidManifest.xml

Should list BLUETOOTH_SCAN, BLUETOOTH_CONNECT, BLUETOOTH_ADVERTISE, plus pre-Android-12 fallbacks.

BLE plugin not loaded means npm install didn't include @capacitor-community/bluetooth-le — re-check package.json dependencies, run npm install, then npx cap sync android.
